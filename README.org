* Status — read this first

Early research project, still pre-1.0.  As of 2026-06-23 it compiles a
useful *integer-C subset* end-to-end to native code (no Rust in the run
path); pointers/structs (M2.3), a libc subset (M3), and the hard-tail C
features needed for real libraries like SQLite/libxml2 (M4-M6) are not
done yet.  See =docs/design/03-roadmap-to-completion.org= for the
milestone ladder and =docs/design/01-02= for the proven substrate.

Breaking changes are expected on most commits.

* Compiling C today (integer-C MVP)

#+begin_src sh
  make cc FILE=examples/demo.c OUT=demo.o   # C -> native ET_REL .o
  cc driver.c demo.o -o prog && ./prog       # link with cc, run
#+end_src

Each C function becomes a C-callable (System V ABI) native symbol.
Working: functions/recursion, =int/long= locals + params, =+ - * / %=,
=& | ^ << >> ~=, comparisons, =&& || !=, =if/else=, =while=, =for=,
early-return guard clauses, assignment (incl. compound), calls, =?:=.
Verified by =test/nelisp-cfront-e2e-test.el= (fib / sumto / gcd /
bitwise compile, link, and run with correct output).

* Purpose

=nelisp-cfront= is the *C front-end* of the NeLisp toolchain.  It takes
C source and lowers it onto the =nelisp-cc= grammar (the s-expression
systems-language that =nelisp= AOT-compiles to native x86_64/arm64).

#+begin_src text
  C source
     │  [front-end: lex → parse → typecheck → C-AST]      (deferred in the spike)
     ▼
  C-AST
     │  [lowering: C-AST → nelisp-cc grammar sexp]         (the spike's subject)
     ▼
  nelisp-cc grammar
     │  nelisp-aot-compile-to-object  (provided by nelisp)
     ▼
  ET_REL .o → link → native execution
#+end_src

Role split, mirroring a real toolchain (cpp / cfront / cc1):

- =nelisp-cc= (in the =nelisp= repo) = *back-end*: grammar → native.
- =nelisp-cfront= (this repo)        = *front-end*: C → grammar.

The name follows Stroustrup's =cfront= (the original C++→C transpiler):
a front-end that lowers onto an existing back-end.

* Why this is plausible (and where it is hard)

The =nelisp-cc= grammar already AOT-compiles a *systems* subset to
native code:

| grammar op | C construct it serves | evidence (nelisp repo) |
|------------+-----------------------+------------------------|
| =ptr-read/write-u8/u64= | =*p=, =p[i]=, =s->f= | =lisp/nelisp-cc-atomic-raw-mem.el= |
| =syscall-direct NR a0..a5= | OS / libc bottom | =lisp/nelisp-cc-alloc-mem.el=, =lisp/nelisp-syscall-table.el= |
| =nl_mmap_alloc/dealloc= | =malloc/free= | =lisp/nelisp-cc-alloc-mem.el= |
| =while= | =for/while= | =lisp/nelisp-cc-sf-while.el= |
| arith / =cmp= / =logand= / =if= / =cond= / =and= | expressions, control | =cc= pipeline |
| f64 ops / int truncation | =double=, narrow ints | =cc-jit-float=, =cc-truncate-int= |

So the lowering target is the same shape as =modernc.org/ccgo= (C→Go on
=unsafe.Pointer= + a re-implemented libc) — except native code is
emitted by =nelisp-cc=, not interpreted.  The genuine cost is the
*front-end* (C99 parse/typecheck/lower) plus a *libc subset* expressed
in the grammar.  See =docs/design/01-c-to-nelisp-cc-lowering.org=.

Known hard tail (not in the spike): =goto=, struct-by-value, varargs,
function pointers, unions/bitfields, =setjmp=.  Also: =nelisp-cc= leaf
ops (=ptr-read-u64= …) currently route through Rust externs; full
pure-elisp end-to-end depends on the =nelisp= Doc 122-128 leaf
migration.

* Spike plan (see Doc 01 for detail)

| stage | proves | grammar exercised |
|-------+--------+-------------------|
| 0 harness ✓ PASS | compile → link → run round-trip (native, no cargo) | (any defun) |
| 1 int-only C ✓ PASS | recursion / if / cmp / arith / narrow-int mask (runtime-free) | if, cmp (< <=), + - *, logand |
| 2 ptr + struct ✓ PASS | real (mmap) memory machine; u32 composed from u8 (gap) | ptr-read/write-u8/u64, nl_mmap_alloc, shl/logior |
| 3 one syscall ✓ PASS | libc/VFS boundary without Rust (write + file roundtrip) | =syscall-direct= |
| 4 local/loop ✓ PASS | gating item: C locals→frame slots + native while, runtime-free | =let=-rt slot, =while=, =setq-local= |

Spike DONE (Stages 0-3 all PASS, runtime-free).  Verdict: FEASIBLE as a
research capability — the native lowering target is proven sufficient for
the systems core (memory, syscalls, arithmetic, C-ABI) with no Rust in
the run path.  Cost is characterised in =docs/design/01= §4 (cost curve):
the gating item is a local-variable / loop lowering strategy that does
NOT use the env-backed grammar =let= (walls 1,2,4), then the C99
front-end, then a libc subset, then hard-tail C (varargs / function
pointers) for SQLite specifically.
