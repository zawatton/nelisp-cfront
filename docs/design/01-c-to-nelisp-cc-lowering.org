* ¬ß0 Summary

=nelisp-cfront= lowers C onto the =nelisp-cc= grammar (the s-expression
systems language =nelisp= AOT-compiles to native x86_64/arm64).  This is
the same architecture as =modernc.org/ccgo= (C‚ÜíGo on =unsafe.Pointer= +
re-implemented libc), except the back-end emits native code instead of
interpreting.

This doc designs a *feasibility spike*: measure whether C semantics fit
the existing grammar, and where the walls are, *before* building a C99
front-end or a libc.  Output = a grammar-coverage / cost-curve report
that decides go/no-go on the full build.

* ¬ß1 Why this is plausible (verified substrate)

The =nelisp-cc= grammar already AOT-compiles a systems subset to native.
Every op below was verified in the =nelisp= source (do not rely on an op
without re-verifying):

| grammar op | C construct | evidence (nelisp repo) |
|------------+-------------+------------------------|
| =ptr-read-u8/u64=, =ptr-write-u8/u64= | =*p=, =p[i]=, =s->f= | =lisp/nelisp-cc-atomic-raw-mem.el= |
| =atomic-fetch-add=, =atomic-compare-exchange= | C11 atomics | =lisp/nelisp-cc-atomic-raw-mem.el= |
| =syscall-direct NR a0..a5= | OS / libc bottom | =lisp/nelisp-cc-alloc-mem.el= |
| =nl_mmap_alloc=, =nl_mmap_dealloc= | =malloc/free= | =lisp/nelisp-cc-alloc-mem.el= |
| syscall nr table (read/write/openat/close/lseek/mmap/munmap‚Ä¶) | libc I/O | =lisp/nelisp-syscall-table.el= |
| =while= | =for/while= | =lisp/nelisp-cc-sf-while.el= |
| arith (=+ - logand= ‚Ä¶), =cmp= (=< > =), =if/cond/and= | expressions, control | =cc= pipeline |
| f64 ops, int truncation | =double=, narrow ints | =cc-jit-float=, =cc-truncate-int= |
| =defun=, =seq= (multi-defun manifest) | functions, TU | grammar |

Toolchain entry: =nelisp-aot-compile-to-object= (=nelisp/lisp/nelisp-aot-compiler.el=
~L16353).  Existing probe runner: =nelisp/scripts/compile-elisp-objects.el=
(registers =:source-var <X>--source= probes; =nelisp-cc-spike-noop= is
the precedent).

De-risk decision: *the spike does not build the C front-end.*  C is
hand-lowered to grammar sexps so the spike isolates the *semantic/runtime
fit* (unknown) from *parsing C* (a standard, known-tractable task).
=src/nelisp-cfront.el= ships the per-construct lowering helpers the
front-end will eventually call.

* ¬ß3 Spike stages

Each stage: a C snippet, its hand-lowered grammar, what it proves, and
the expected wall.

** Stage 0 ‚Äî harness (DONE ‚Äî VERIFIED 2026-06-22)

Establish compile ‚Üí link ‚Üí run before any C semantics.

#+begin_src emacs-lisp
  (defconst stage0--source '(defun nelisp_cfront_stage0_add (a b) (+ a b)))
  ;; nelisp-aot-compile-to-object ‚Üí stage0_add.o ‚Üí cc links a C driver
  ;; (extern long f(long,long)) ‚Üí ./stage0 ‚Üí add(3,4) prints 7, exit 0
#+end_src

RESULT: =make stage0= PASS.  =add(3,4) = 7=, exit 0.  Round-trip proven
end-to-end with *no cargo/Rust in the run path* ‚Äî a C driver links the
=.o= directly via =cc=.

Verified facts (locked for later stages):
- API: =(nelisp-aot-compile-to-object SEXP FILE-PATH &key (arch 'x86_64)
  (format 'elf))= ‚Äî SEXP is the FIRST arg.
- The emitted defun is a GLOBAL =STT_FUNC= (=nm= shows
  =T nelisp_cfront_stage0_add=), C-callable via System V AMD64 ABI
  (i64 params in rdi/rsi, i64 return in rax) ‚Äî matches =extern long
  f(long,long)=.
- Object is =ET_REL= x86_64 ELF; =cc= (Debian 14.2.0) links it cleanly.
- Bodies must not contain strings (v1 object mode forbids =write= /
  rodata) ‚Äî Stage 1+ keeps that constraint until the rodata path lands.

Implemented in =spike/stage0-harness.el= (=nelisp-cfront-stage0-run=);
artifacts land in =spike/out/= (git-ignored).

** Stage 1 ‚Äî int-only C (DONE ‚Äî VERIFIED 2026-06-22)

#+begin_src c
  long sum(long n){ long s=0; for(long i=0;i<n;i++) s+=i; return s; }
  long narrow(long x){ return (uint8_t)x; }
  long fact(long n){ return n<=1 ? 1 : n*fact(n-1); }
#+end_src

RESULT: =make stage1= PASS.  =sum(10)=45 narrow(300)=44 fact(5)=120=.
The object is *runtime-free* ‚Äî =nm= shows only =T= symbols, no =U=
undefined externs; it links with a bare C driver, no nelisp runtime.

*** WALL FOUND: grammar =let=/=setq= are environment-backed

The first hand-lowering used the natural iterative form:

#+begin_src emacs-lisp
  (let ((s 0) (i 0))
    (while (< i n) (seq (setq s (+ s i)) (setq i (+ i 1)))) s)
#+end_src

It *compiled* but *failed to link*: undefined references to
=nl_alloc_symbol= and =nelisp_env_set_value=.  So grammar =let=/=setq=
do NOT lower to raw machine locals ‚Äî they bind into the nelisp *runtime
environment* (symbol alloc + env set).  Consequences for C lowering:

- C locals must NOT be lowered to grammar =let=/=setq= if a runtime-free
  object is wanted.  Lower them as *function parameters threaded through
  recursion* (params are raw i64 registers ‚Äî proven by stage0 and the
  Stage 1 recursion form), i.e. an SSA / param-passing style in the
  front-end.
- =nelisp-cc= has *no TCO* (confirmed in =nelisp-cc-sf-while.el=
  commentary): recursion depth = iteration count, so large loops
  overflow the native stack (~650k frames in nelisp's own measurement).
  A real C front-end must either (a) compile loops to a host =while=
  form (which currently drags in the env runtime), or (b) emit a native
  loop without env-backed mutation ‚Äî a candidate *new grammar op*
  (register-local loop), or (c) accept linking the nelisp runtime for
  functions that use env locals.

Confirmed runtime-free grammar ops (link with no externs): =if=, =<=,
=<==, =+=, =-=, =*=, =logand=, self-recursion, multi/forward-ref defuns
via =(seq ...)=.

Remaining Stage 1 wall candidates (not yet exercised): signed‚Üîunsigned
division/shift semantics, integer-promotion rules, =u16/u32= widths.

** Stage 2 ‚Äî pointer + memory + struct (DONE ‚Äî VERIFIED 2026-06-22)

Three probes, one runtime-free object (=nm= shows no =U= externs):

#+begin_src c
  // A. struct field rw on a caller pointer:  rw(p){p->x=5;p->y=99;return p->x+p->y;}  -> 104
  // B. self-allocated heap:  heap(){P*p=malloc(16);p->x=7;p->y=35;r=...;free(p);return r;} -> 42
  // C. u32 read composed from 4 bytes (LE):  read_u32_le(buf) -> 0x12345678
#+end_src

#+begin_src emacs-lisp
  (defun nelisp_cfront_stage2_struct_rw (p)
    (seq (ptr-write-u64 p 0 5) (ptr-write-u64 p 8 99)
         (+ (ptr-read-u64 p 0) (ptr-read-u64 p 8))))
  ;; heap: pointer threaded through PARAMS (let is env-backed, Stage 1);
  ;; reuses nelisp's grammar-level nl_mmap_alloc / nl_mmap_dealloc
  ;; (syscall-direct mmap/munmap ‚Äî splices in, stays runtime-free).
  (defun nelisp_cfront_stage2_read_u32_le (p off)
    (logior (ptr-read-u8 p off)
      (logior (shl (ptr-read-u8 p (+ off 1)) 8)
        (logior (shl (ptr-read-u8 p (+ off 2)) 16)
                (shl (ptr-read-u8 p (+ off 3)) 24)))))
#+end_src

RESULT: =make stage2= PASS. =struct_rw=104 heap=42 read_u32_le=0x12345678=.
The "memory machine" is REAL native memory (not emulation), and the
grammar-level mmap allocator composes into a downstream object with zero
externs.

Confirmed: =ptr-read/write-u8/u64= on real memory; pointers are plain
i64 in registers (like any param); =nl_mmap_alloc/dealloc= reusable and
runtime-free; =shl / logior / logand= available (production AOT
nelisp-cc-jit-secure-hash).

*** GAP: no single u16/u32 load/store op (only u8 / u64)

A =u32= read takes 4 =ptr-read-u8= + 3 =shl= + 3 =logior= (a =u16= takes
2+1+1).  Functionally complete and matches C's little-endian layout
(verified against C's =*(uint32_t*)=), but a native =ptr-read/write-u16/u32=
op would cut code size and cost ‚Äî a worthwhile grammar addition for any
real C target (SQLite is full of u32 fields).  -> cost-curve ¬ß4.

Carried forward: pointers/structs use params not =let= (env-backed);
alignment not stress-tested (malloc/16-aligned only); pointer-to-pointer
and struct-by-value not yet exercised.

** Stage 3 ‚Äî one syscall / file I/O (DONE ‚Äî VERIFIED 2026-06-22)

Thin =syscall-direct= wrappers (write/read/openat/close); the C driver
orchestrates.  Buffers + path come from the driver (the v1 object mode
forbids string literals in grammar bodies).

#+begin_src emacs-lisp
  (defun nelisp_cfront_stage3_write (fd buf n) (syscall-direct 1 fd buf n 0 0 0))
  (defun nelisp_cfront_stage3_read  (fd buf n) (syscall-direct 0 fd buf n 0 0 0))
  ;; openat(AT_FDCWD=-100, path, O_WRONLY|O_CREAT|O_TRUNC=577, 0644=420)
  (defun nelisp_cfront_stage3_openat_wr (path) (syscall-direct 257 -100 path 577 420 0 0))
  (defun nelisp_cfront_stage3_openat_rd (path) (syscall-direct 257 -100 path 0 0 0 0))
  (defun nelisp_cfront_stage3_close (fd) (syscall-direct 3 fd 0 0 0 0 0))
#+end_src

RESULT: =make stage3= PASS (runtime-free object).  =write(1,"hi\n",3)=
prints =hi=; the file round-trip openat‚Üíwrite‚Üíclose / openat‚Üíread‚Üíclose
created =/tmp/nelisp_cfront_stage3.txt= and read its bytes back equal to
what was written.  *The libc/VFS bottom works with zero Rust.*

*** Constraint observed (same family as the Stage 1 wall)

Orchestrating multiple syscalls *inside one grammar function* needs to
hold an fd in a local across calls ‚Äî which is the env-backed-=let= /
no-=TCO= wall.  So the spike orchestrates from the driver and keeps each
wrapper a single inline syscall.  A real front-end must own local
liveness (SSA / explicit stack frame), not lean on grammar =let=.

Deferred libc concerns (not blockers, just unimplemented): =errno=
convention is "negative return = -errno" (no global, no =perror=);
buffered stdio; =string=‚ÜîC-buffer marshaling (here the driver supplies
buffers).

All four spike stages PASS, each producing a *runtime-free* object

** ¬ß4.2 Walls found (cost to clear)

1. *=let=/=setq= are env-backed* ‚Äî they emit =nl_alloc_symbol= /
   =nelisp_env_set_value=, needing the nelisp runtime.  C locals must be
   lowered as *SSA / param-threaded* (or to a future register-local op),
   NOT to grammar =let=.  [Stage 1]  ‚Äî *the #1 design decision; walls 2
   and 4 are the same root.*
2. *No TCO* ‚Äî recursion depth = iteration count; big loops overflow the
   native stack.  Needs a *register-local loop op* (new grammar op) or
   front-end SSA-with-backedge.  [Stage 1]
3. *No single =u16/u32= memory op* ‚Äî composable from u8 but a native
   =ptr-read/write-u16/u32= would cut size/cost (SQLite is u32-heavy).
   [Stage 2]
4. *Multi-statement orchestration needs locals across calls* ‚Äî same root
   as #1.  Front-end owns local liveness.  [Stage 3]
5. *v1 object mode: no strings / rodata* ‚Äî string literals can't be
   embedded; data comes from the caller until a rodata path lands.  [0,3]

|-----------+-------+------|
| =memcpy/memset/memmove= | ‚ë£ write in grammar | trivial over =ptr-*= |
| =malloc/free/realloc= | ‚ë£ (have mmap base) | need slab + realloc on =nl_mmap_*= |
| =str*= (=strlen/strcmp/‚Ä¶=) | ‚ë£ write in grammar | straightforward over =ptr-read-u8= |
| =printf=-family | ‚ë£ + ‚ë¢ varargs | *hard tail* ‚Äî varargs unproven |

| =errno=/=perror= | ‚ë£ convention | negative-return = -errno; no global yet |

** ¬ß4.4 Hard-tail C (Vacietis-class; SQLite uses these)

=goto=, struct-by-value args/returns, *varargs* (printf ‚Äî heavy),
*function pointers* (VDBE dispatch ‚Äî heavy), unions/bitfields, =setjmp=.
These are not exercised by the spike and are the genuine long-pole for
SQLite specifically.

** ¬ß4.5 Go / no-go

FEASIBLE as a research capability: the native lowering target is proven
sufficient for the systems core, end-to-end, with no Rust in the run
path.  NOT cheap.  Cost concentrates in, in order:

1. *Local-variable / loop lowering strategy* (walls 1,2,4) ‚Äî must land
   first; everything downstream depends on it.  Likely = front-end SSA +
   one or two new grammar ops (register-local loop; u16/u32 mem).
2. *C99 front-end* (lex/parse/typecheck/lower) ‚Äî the bulk, but standard.
3. *libc-in-grammar subset* (¬ß4.3) ‚Äî substantial but mechanical except
   varargs.
4. *Hard-tail C* (¬ß4.4) ‚Äî the real long-pole if/when SQLite is the target.

Recommended next step if continuing: resolve item 1 (a register-local /
SSA loop that lowers a C =for= to native WITHOUT env-backed =let=),
since it gates the front-end design.  SQLite remains a distant target
behind items 2-4.

* ¬ß5 Honest residual risks (NOT in the spike)

- Hard-tail C (the features Vacietis could not do, that SQLite uses):
  =goto=, struct-by-value args/returns, varargs (printf-family ‚Äî heavy in
  SQLite), function pointers (VDBE), unions/bitfields, =setjmp=.
- Leaf-op Rust dependency: =ptr-read-u64= etc. currently route through
  Rust externs (=nl_ptr_read_u64=).  Full pure-elisp end-to-end is gated
  on the =nelisp= Doc 122-128 leaf migration.  The =nelisp-cfront=
  front-end is pure elisp regardless.
