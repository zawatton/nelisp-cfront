* §3 Prioritised backlog for M5 (front-end first, by demand × blocking)

Tier 1 — *parse blockers that appear constantly* (without these, parsing
stops almost immediately):
1. =__attribute__= / =__extension__= / GCC builtin types — recognise &
   skip in the lexer/parser (=__builtin_va_list= → treat as an opaque type).
2. casts =(type)expr= — parse; lower as (mostly) identity in the untyped
   i64 model.
3. multi-declarator (=int a, b;=) and declaration initializers.

Tier 2 — *high-demand language features*:
4. aggregate + designated initializers (1426+149) — parse + lower to a
   sequence of field/element stores.
5. =float=/=double= (1434) — lower onto the existing nelisp-cc f64 grammar
   ops (=f64-add= etc., =extern-call-f64=); type-track float vs int.
6. =switch=/=case= (1209 cases) — lower to an if-chain or jump (flag-guarded
   like break).
7. =do=…=while=, =enum=.

Tier 3 — *the genuinely hard / upstream items* (Doc 04):
8. robust =goto= (719) — multi-label / backward needs a relooper.
9. =varargs= (153, printf-family) — needs the variadic ABI in *nelisp-cc*
   (upstream).
10. struct-by-value locals & =&local= — need an =alloca= grammar op in
    *nelisp-cc* (upstream).

* §6 Float lowering — SOLVED via "soft-float in the i64 world"

** §6.1 The blocker: nelisp-cc's f64 grammar is flat-only

Surveying the upstream f64 grammar (=f64-add/sub/mul/div=, =f64-lt/gt/le/ge=,
=i64-to-f64=, =f64-to-i64-trunc=, =bits-to-f64=, =f64-call=, =extern-call-f64=)
found it is MVP "flat-only", *not* enough for real C double code:

- *Nested f64 rejected* — =--emit-f64-leaf-into= signals
  =:nested-f64-binop-needs-doc-112=; an f64-binop operand must be a leaf
  (=ref :class f64= / =bits-to-f64= / =i64-to-f64= / =f64-call=).  So
  =(f64-add (f64-mul a b) c)= does not compile.
- *No f64-class local* — =let-rt= bindings are always =:class gp=; there
  is no double stack slot, store, or reload.
- *No mixed params* — SysV defuns require a uniform param class, so
  =double f(double, int)= cannot even be expressed.

i.e. native f64 was the same kind of *upstream* dependency as varargs /
alloca (needs Doc 112 xmm-spill), NOT a quick cfront-side gap.

** §6.2 The fix: carry =double= as its i64 bit pattern

A C =double=/=float= is represented as its raw IEEE-754 *64-bit pattern
in a gp (i64) register/slot* — exactly the value class cfront already
supports everywhere (frame slots, =setq=, calls, returns).  Each float op
lowers to a flat per-object helper call:

| C            | lowering                                    | result    |
|--------------+---------------------------------------------+-----------|
| =a + b=      | =(nelisp_cfront__dadd A-bits B-bits)=       | bits      |
| =a < b=      | =(nelisp_cfront__dlt  A-bits B-bits)=       | i64 0/1   |
| =a == b=     | =(nelisp_cfront__deq ..)= (= dle & dge)     | i64 0/1   |
| =-a=         | =(logxor A-bits 0x8000…)= (sign-bit flip)   | bits      |
| =(double)i=  | =(nelisp_cfront__i2d I)=                     | bits      |
| =(long)d=    | =(nelisp_cfront__d2i D-bits)= (truncate)    | int       |
| =3.14=       | i64 literal = IEEE-754 bits (frexp encoder) | bits      |

Each helper is one *flat* f64 op wrapped in =bits-to-f64= / =f64-bits=,
so nesting / locals / mixed params all "just work" — every double is an
ordinary i64 the rest of the pipeline already handles.  No Doc 112,
no xmm spill, no f64 =let= slots.

** §6.3 The one upstream primitive: =f64-bits=

The only missing piece was the inverse of =bits-to-f64=: a grammar op that
takes an f64-class result and reinterprets its raw bits as a gp i64.
Added =(f64-bits F64-EXPR)= (opcode 100) to =nelisp-aot-compiler.el= —
emits the f64 expr into xmm0 (binop or leaf) then =MOVQ rax, xmm0= (the
asm primitive =movq-r64-xmm= already existed).  Both x86_64 + aarch64
wired; 2 byte-level tests added; upstream f64 + aot suites stay green
(125/125 + 169/169).

*ABI note*: a =double=-returning cfront function returns the bits in rax

* §7 Enum constants — DONE

The parser now *parses* an enum body (instead of skipping it) and
registers each =NAME -> value= into =--enum-consts= with C semantics:
auto-increment from 0, explicit =NAME = CONST=, and constant-expression
initializers folded by =--const-eval= (literals, =+ - * / % << >> & | ^=,
comparisons, =&& ||=, =~ ! -=, casts, ternary, and references to prior
enum constants).  =parse-primary= folds a reference to a registered
constant directly into =(int VALUE)=, so lowering needs no special case.

Verified end-to-end (=test/nelisp-cfront-enum-test.el=): a program using
bare + tagged + flag enums (=F2 = 1<<1=, =FALL = F1|F2|F3=) compiles and
runs natively — =pick(1)=2 pick(5)=101 vals()=210 colors()=222
flags()=7=.  =make gap= unchanged at line 24756 (no parse regression:
all enums in the parsed prefix fold cleanly).  Suite 57/57 green.

* §8 Float call-arg coercion — DONE

=lower.el= now tracks a =--func-params= table (NAME -> positional list of
param types, built from =func=/=proto= top-levels; the =(void)= marker is
dropped, unnamed prototype params kept so positions stay aligned).  A
direct call coerces each argument to its callee's param class via
=--coerce= (=i2d= for an int arg into a =double= param, =d2i= for the
reverse); arguments beyond the param list (varargs) pass through
uncoerced, as do fn-pointer calls (param types not tracked there).

Verified end-to-end (=nelisp-cfront-float-test.el::...-arg-coercion-e2e=):
=fromint(7)= → =scale(7.0, 2.0) = 14.0= (the int =7= is lifted to
double-bits before the call), =half(10.0) = 5.0=, =trunc_scaled(2.5) = 7=
(int literal =3= coerced to =3.0=).  Suite 58/58 green; =make gap=
unchanged at 24756.

With §6 (arithmetic/locals/conversions), §8 (call args), the only
remaining float gap is the *external* SysV =double= ABI (xmm0 returns/
args for non-cfront callers) — a thin =extern-call-f64= + =f64-bits=
shim, deferred until an external double boundary actually needs it.

* §9 Aggregate locals & address-of-local — DONE (upstream `frame-alloc')

** §9.1 Upstream keystone

nelisp-cc had no way to address frame memory (struct-by-value locals and
=&local= were blocked).  Added =(frame-alloc NBYTES)= (opcode 101): it
reserves =ceil(NBYTES/8)= slots from the defun's frame pool (same counter
as =let-rt=, so the prologue's =sub rsp= covers it) and returns the
block's lowest address via =mov rax, rbp; sub rax, K= (imm32, so large
blocks bypass the disp8 slot range).  Stable across re-eval (slots fixed
at parse) = a C local, not a dynamic =alloca=.  x86_64 SysV only.

** §9.2 cfront memory-backed locals

A local is *memory-backed* when its type is an array / struct-by-value,
or its address is taken (=&x=, found by a full-tree walk of the body).
=lower.el= binds such a local's slot to =(frame-alloc SIZE)= (the slot
holds the block *address*); then:
- an aggregate var *decays* to that address (so =a[i]=, =&a=, =s.f= work);
- an address-taken scalar is read / written / =&='d *through* the
  pointer (load / store / the address itself);
- =&lvalue= generalised: =&x= / =&a[i]= / =&s.f= / =&*p= all lower to the
  computed address with no value load.

Supporting fixes: =type-elem= (array indexing element type — arrays decay,
=type-pointee= only handled pointers); array dimensions are now
constant-folded by the parser (=--fold-dim=, tolerant: VLA / =sizeof=
(struct) fall back to an unknown =t= dimension so parsing never stalls;
scalar/pointer =sizeof= folds); fixed a latent =--strip-array= plist
corruption bug exposed once array element sizes were actually computed.

Deferred follow-ons: address-taken *params* (need an entry register

** §9.4 Varargs (the other half of "alloca + varargs")

=frame-alloc= is also the foundation for the varargs *register save area*,
but full SysV variadic support is a *separate, larger* ABI effort:
callee =va_start= must spill the incoming GP (rdi-r9) + XMM (xmm0-7)
arg registers into a save area and build the 24-byte =va_list= struct
(gp_offset / fp_offset / overflow_arg_area / reg_save_area); =va_arg=
walks it per type class; the caller must set =AL= (vector-count) and
pass the stack-arg tail.  Scoped as the next upstream milestone — the
=frame-alloc= keystone landed here unblocks its save-area piece.

** §10.1 =:unknown-struct= crushed (1065 → 93; coverage 36% → 55%)

Root cause was *anonymous / inline-defined struct & union members*: the
type machinery only resolved a struct by *tag name*, so a member like
glibc =__mbstate_t='s anonymous =union {...} __value= (stored as
=:base struct :struct nil :fields (...)=) failed =type-struct nil=, which
made =type-build-structs= skip the whole enclosing struct — and every
function touching it then failed =:unknown-struct=.  Just 224 of 267
structs laid out.

Fix: =type--resolve-struct= computes a struct/union's layout from its
inline =:fields= when present, else by tag.  Wired into =type-size=,
=type-align=, and a new =type-field-ty= (member access by type, not tag).
Result: *266/267 structs lay out*; =:unknown-struct= dropped 1065 → 362
(type-size/align) → 93 (type-field-ty); coverage 36.2% → 50.3% → 55.1%
(1,318 / 2,392 functions).

The residual 93 are mostly *global variables* of struct type (e.g.
=sqlite3Stat.nowValue[op]= in =sqlite3StatusValue=): cfront defers
globals, so a global isn't in =tenv=, types as =long=, and member access
fails.  Global-variable support is a separate feature.  (The last
un-laid-out struct, =RowSetChunk=, has a =sizeof(struct)= array
dimension — needs parse-deferred dim evaluation.)

After this, the dominant bucket is =:unsupported-load-width= (388 —
struct-by-value loads), the natural next lowering target.

** §10.2 =:unsupported-load-width= crushed (388 → 0; coverage 55% → 65%)

The width came from trying to *load a whole aggregate value*.  Root cause
was that an array-/struct-typed member, index, or deref in rvalue context
went through =--load-lvalue=, e.g. =db->aLimit[0]= where =aLimit= is
=int aLimit[12]=: lowering =db->aLimit= tried to load the 48-byte array
before indexing it.

Fix — the same "aggregate decays to its address" rule already used for
*vars*, now applied to member / arrow / index / deref via a new
=--rvalue=: an aggregate lvalue yields its *address* (which =a[i]=, =s.f=,
=&=, struct copy all consume), a scalar is loaded.  Plus genuine struct
*copy* (=b = a=, =struct T x = lvalue;=) now lowers to a word-wise
=--copy-bytes= (u64/u32/u16/u8 at constant offsets) instead of a scalar
load/store.

Result: =:unsupported-load-width= 388 → *0*; coverage 55.2% → *64.9%*
(1,553 / 2,392 functions).  Verified e2e: =getlim/sumcfg= (array-member
indexing through a pointer) and =copypt= (=b = a= struct copy) =30 60 42=.

The dominant bucket is now =:string-literal-unsupported= (244 — needs an
object-mode rodata string pool), then =:incdec-needs-var= (174 — lvalue
=++=/=--=).

** §10.3 =:incdec-needs-var= crushed (174 → 0; coverage 65% → 71%)

=++=/=--= only handled plain var targets.  Generalised by *desugaring*
=TARGET++= to =TARGET = TARGET + 1= and reusing =--assign= / =--binop=:
this covers every lvalue (=p->f=, =a[i]=, =*p=, bitfield, address-taken
scalar) and — because the =+=/=-= flows through =--binop= — also fixes
pointer =++= (advances by the pointee size) and float =++= (adds =1.0=),
which the old var-only path got wrong.  (A side-effecting lvalue
subexpression like =a[i++]++= is re-read and not modelled — rare.)

Result: =:incdec-needs-var= 174 → *0*; coverage 64.9% → *70.8%*
(1,693 / 2,392 functions).  Verified e2e: =memberinc/arrinc/derefinc=
(=c->n++= / =a[i]++= / =(*p)++=), =ptrnext= (=p++= = scaled, lands on the
next element), =predec= (=--x=) → =42 6 100 20 49=.

Dominant bucket now =:string-literal-unsupported= (263 — object-mode
rodata string pool), then =:early-return-in-loop= (136) and
=:unknown-struct= (128 — globals).

** §10.4 =:early-return-in-loop= crushed (136 → 0; coverage 71% → 75%)

The single-exit detector (=--return-in-loop-p=) only flagged returns
inside *loops*, but a return is lowered via =--effect= (and so needs the
return flag) in three shapes: loops, =switch= case bodies, AND a
partially-returning =if/else-if= chain followed by more statements (e.g.
=parseDateOrTime='s =if(..)return; else if(..)return; ... return 1;= —
the chain isn't all-returning, so =--stmts-tail= lowers it via effect).

Fix: replace the loop-only check with =--stmts-tail-effect-return-p= /
=--tail-effect-return-p=, which *precisely mirror* the tail-lowering rules
(=--stmts-tail= / =--tail=) and report any return that would route
through =--effect= — subsuming loops, switches, and if-chains.  Also
guard switch case groups by all active exit flags (so =return= in a case
bypasses later cases).

Result: =:early-return-in-loop= 136 → *0*; coverage 70.8% → *75.2%*
(1,798 / 2,392 functions).  Verified e2e (=long= params): a
partially-returning if/else-if chain (=classify=), =switch= returns
(=sw_ret=), and a loop return (=loop_ret=) → =-1 0 1 2 129 2 -1=.

*** §10.4.1 narrow-int ABI sign-extension — FIXED (§10.5)

** §10.5 Narrow-int width correctness (char/short/int)

A correctness fix (not a coverage bucket — =lower-gap= only checks that
lowering *succeeds*, so this was invisible there).  cfront carries every
value as i64; narrow ints need explicit sign/zero extension at two
boundaries:

1. *Function entry* — SysV passes a narrow int arg in the low bits of a
   64-bit register with the high bits unspecified (gcc =mov edi,-5=
   *zero*-extends =rdi= to =0xFFFFFFFB=).  Each narrow-int param is now
   re-normalized at entry: =(setq p (sar (shl p K) K))= for signed
   (K = 64 − 8·width), =(logand p MASK)= for unsigned.
2. *Memory load* — =--load-w= zero-extends, so a SIGNED narrow field is
   sign-extended after the load (=(sar (shl g K) K)=); unsigned is already
   exact from the masked byte load.

Helpers: =--narrow-int-width= (1/2/4 for char/short/int, else nil) and
=--normalize-narrow=.  Verified e2e: =isneg(-5)=1=, =cmp2(-3,2)=1=,
signed-char/short/int fields read back =-3 / -300 / -100000=, unsigned
char =200=, and =classify= with an =int= param now branches correctly
(=-888=).  Coverage unchanged (75.2%); suite 64/64.

Still simplified (rare): narrow *arithmetic* overflow (=int+int= wraps at
32 bits) is computed in 64-bit, and a narrow *local* assigned a wider
value isn't truncated in its value slot — both only matter on actual
overflow / out-of-range stores, deferred.

** §10.6 =:deref-non-pointer= diagnosis → global variables (Doc 06)

Diagnosed (not a type-inference micro-gap): every sampled =:deref-non-pointer=
function fails *indexing a global array* — =sqlite3UpperToLower[c]=,
=sqlite3CtypeMap[c]=, =sqlite3Utf8Trans1=, =sqlite3azCompileOpt[0]=.  The
global isn't in =tenv=, types as =long=, and =a[i]= / =*p= signals.

This is the same root as the global share of =:unknown-struct= and all of
=:string-literal= — together ≈480 of the ≈594 remaining failures need
*global / static data in cfront's ET_REL object*, addressable from =.text=.

Feasibility verified: the ELF writer (=nelisp-elf-write.el=) already has
=.rodata=/=.data=/=.bss=, =.symtab=, and =abs64=/=pc32= relocations; but
the AOT *compiler's* object path emits no data, and the
=static-imm32-table= lever is executable-mode-only (bakes an absolute
vaddr → fails in ET_REL).  So global support is the next *milestone*
(upstream object-mode data+reloc op, then cfront global emission), not a
quick bucket fix.  Full design + layered plan in *Doc 06*.

* §12 =:deref-non-pointer= sweep: 91% → 94.5% (2026-06-24)

Precise per-expr diagnosis (advise =type-of=, capture the innermost expr
that signals) split the 66 =:deref-non-pointer= into three real causes:

- =*p++= / =*++p= (≈23): =type-of= had no =pre=/=post= arm, so the
  post-increment typed as =long= and the deref failed.  Added

* §13 =:non-constant-array-size= → function-static locals (94.5% → 95.6%)

Precise diagnosis (advise =type--const=): all 38 actionable cases had a
=:array t= (implicit =[]=) dimension reaching =type-size=.  The trigger
was *function-=static= locals* — =static LogEst a[] = {0,2,3,5,6,7,8,9};=
inside =sqlite3LogEst=, a read-only lookup table.  The parser *discarded*
=static=, so it was treated as an automatic local: frame-alloc needs the
full array size, but the implicit =[]= has no const dimension.

Fix (cfront-only): the parser tags =static= as =:storage static= on the
decl type; the lowerer *lifts* each function-static local to a uniquely-
named module global (=nlcf_sl_N_NAME=) — exactly the global-data path
(rodata for a read-only table, =.data=/=.bss= for a mutable scalar, the
implicit =[]= resolved from the initializer).  References within the
function lower via =(data-addr SYM)= (=--static-locals= map), and the
static decl emits no runtime initializer.

Result: =:non-constant-array-size= 41 → 13, coverage 94.5% → *95.6%*
(2286 / 2392).  Remaining 13 are genuine VLAs / non-constant dims.  Top
buckets now: =:unknown-struct= 47 (incomplete-type tail), =:addr-of-non-
memory-var= 27 (=&param= / =&otherGlobal=).

* §14 =:addr-of-non-memory-var= → address-taken params (95.6% → 96.2%)

The 27 split ~half =&param= (=&x= on a scalar parameter) and ~half
=&global= on globals with a *designated* initializer that =--lay= skips
(=.field=='-style io-method tables — deferred).

Fixed =&param=: an address-taken scalar param is spilled to a frame block
at entry.  The value arrives in a register under a =nlcf_pin_NAME= temp;
=--func= frame-allocs a block bound to the param's =--lvar= slot, stores
the (narrow-normalized) incoming value into it, and adds the param to
=--mem-vars= so refs load through the block and =&p= returns its address —
exactly the address-taken-local path, extended to params.

Result: =:addr-of-non-memory-var= 27 → 11, coverage 95.6% → *96.2%*
(2301 / 2392).  The residual 11 are =&global= on designated-initializer
globals (need parser designator preservation + by-name =--lay= placement).

* §15 struct-table completeness: 96.2% → 98.0% (2026-06-24)

Chasing the `&global' tail of `:addr-of-non-memory-var' (io-method tables)
revealed those globals were *positional*, not designated — and the real
blocker was the *struct table*.  `build-structs' only scanned top-level
`struct-def' / `typedef', so a tag defined inline *inside another struct's
members* (=struct ExprList_item= nested in =struct ExprList=) or *at a
sibling global decl* (=struct sqlite3PrngType= defined at =sqlite3Prng=,
referenced by =sqlite3SavedPrng=) was never registered — so any global or
expression of that type failed (=:unknown-struct= / unsizeable global ->
=&= / member access).

Fix (=nelisp-cfront-type=): a recursive =--scan-inline= registers every
inline struct/union definition reachable from a type — nested in members,
and at every =global= / =func= return+param type — innermost first.

Result: =:unknown-struct= 47 → *5*, =:addr-of-non-memory-var= 11 → 6,
coverage 96.2% → *98.0%* (2345 / 2392).  Remaining 47 are now a small
mixed tail: =:non-constant-array-size= 13 (VLAs), =:deref-non-pointer= 8,
=:varargs= 7, =:unknown-field= 7, =:addr-of= 6, =:unknown-struct= 5.

* §16 :unknown-field/:unknown-struct tail (98.0% → 98.1%) + sizeof fixes

Precise diagnosis (advice on `type-struct'/`type-field-ty', then advice-
free replication — the advice itself over-reported) split the 12 cases:

- =:unknown-struct "RowSetChunk"= (sqlite3RowSetClear, rowSetEntryAlloc):
  the struct has an array field dimensioned by =sizeof(struct RowSetEntry)=
  — the parser cannot fold a struct =sizeof= (needs the layout table), so
  the dim became =t= and the struct's own layout failed.  Fixed: the
  parser now *retains* the dimension expression (=--fold-dim=), and a

  using the struct table.  =build-structs= also scans function-body local
  decls (=--scan-body=) for inline struct tags.
- =--sizeof= bug (orthogonal, found here): =sizeof(TYPE)= returned a flat
  8 for struct/array/float.  Now routed through =type-size= (struct layout,
  =sizeof'-dim arrays, float=4).
- =:unknown-field= (=Column.iMem=, =FuncDef.data=, …) and the remaining

  local name redeclared with a different struct-pointer type in sibling

Result: =:unknown-struct= 5 → 3, coverage 98.0% → *98.1%* (2347 / 2392),
plus correct =sizeof(struct/array/float)=.  Remaining =:unknown-field= 7 /

* §17 lexical block scoping (rename shadowed locals) — 98.1% → 98.2%

The `:unknown-field' bucket (7) was *type confusion*: cfront keeps locals
in one flat per-function tenv, so a name redeclared with a different type
in sibling scopes (`LookasideSlot *p' vs `HashElem *p' in two `switch'
cases of `sqlite3_db_status') resolved to one type and `p->field'
mis-resolved — silently mis-typed even in functions that happened to lower.

Fix (`nelisp-cfront-lower--rename-shadowed', run first in `--func'): a
pre-pass renames every declaration of a *multiply-declared* name (and its
references, resolved by walking block scope) to a unique slot; the flat
tenv/slot space is then correct.  Functions with no redeclared name are
returned unchanged (zero impact on the ~2300 already-OK functions).

Result: `:unknown-field' 7 → *0* (the type confusion is gone), coverage
98.1% → 98.2% (2348 / 2392).  The net coverage move is small because most
of those 7 functions, once correctly typed, hit a *different* remaining
gap (an anonymous `:unknown-struct nil' or `&' case) — but the latent
mis-typing is fixed for every shadowing function, not just the failing
ones.  Remaining tail: `:non-constant-array-size' 12 (VLAs),
`:unknown-struct' 9 (anonymous tag-nil structs + a function-local
`Sublist' layout), `:addr-of' 8, `:deref-non-pointer' 7, `:varargs' 7.
