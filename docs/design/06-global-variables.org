#+TITLE: Doc 06 — Global variable support (M5 lowering milestone)
#+AUTHOR: zawatton + Claude
#+DATE: 2026-06-23
#+STATUS: design + feasibility verified; implementation is a multi-step milestone

* §0 Why

After the M5 lowering sweep (Doc 05 §10, coverage 36% → 75%), the
=lower-gap= backlog is dominated by *global variables*:

| bucket | count | global-related |
|--------+-------+----------------|
| =:string-literal-unsupported= | 276 | yes (rodata strings) |
| =:unknown-struct= | 134 | ~100 (global struct vars) |
| =:deref-non-pointer= | 110 | yes (global array index/deref) |

That is ≈480 of the ≈594 remaining lowering failures.  All three share
one missing capability: *static/global data in cfront's ET_REL object
output, addressable from =.text=*.

Diagnosis evidence (=:deref-non-pointer=): every sampled function
(=sqlite3StrICmp=, =sqlite3Utf8Read=, =getDigits=, =sqlite3CompileOptions=,
…) fails indexing a *global array* — =sqlite3UpperToLower[c]=,
=sqlite3CtypeMap[c]=, =sqlite3Utf8Trans1[...]=, =sqlite3azCompileOpt[0]=.
The global isn't in =tenv=, types as =long=, and =a[i]= /=*p= then
signals =:deref-non-pointer=.

* §1 Why it is NOT a quick fix

- cfront skips =global= top-levels entirely (=lower-program=), and its
  object mode emits *no* data — only =.text= (the "v1 object mode forbids
  rodata" limitation).
- Type tracking alone is insufficient: even with a global's type known, a
  reference needs the global's *address*, which in an ET_REL object is a
  *relocation* against a data symbol — not an absolute value.
- The obvious upstream lever, =static-imm32-table-define= /
  =static-imm32-table-lookup=, is *executable-mode only*: it bakes an
  absolute =.rodata= vaddr (=mov rdi, <vaddr>=), which is unknown in a
  relocatable object → it signals =:static-imm32-table-vaddr-missing= when
  driven through =nelisp-aot-compile-to-object=.  Verified 2026-06-23.

* §2 Feasibility — the ELF writer already has the primitives

=lisp/nelisp-elf-write.el= supports everything at the file level:
- =.rodata= / =.data= / =.bss= (NOBITS) sections,
- =.symtab= with =st_value= (defined local/global symbols),
- relocations: =R_X86_64_64= (=abs64=), =R_X86_64_PC32= (=pc32=),
  =R_X86_64_PLT32= (=plt32=); reloc-type map in =--reloc-type-code=.

So the *file format* can express "a data blob with a local symbol, and a
=.text= PC32 reference to it".  The gap is purely in the AOT *compiler*
pipeline: the ET_REL emit path does not currently (a) place
caller-provided data blobs into =.rodata=/=.data=/=.bss= with a symbol,
nor (b) expose a grammar op that emits a same-unit data-symbol address
reference (the existing =data-addr= is for *external* symbols only).

* §3 Plan (layered; re-measure with =make lower-gap= after each step)

** Step A — upstream keystone: same-unit data definition + address  [DONE 2026-06-24]
Added to =nelisp-aot-compiler.el= (ET_REL path):
- =(data-blob NAME BYTES SECTION)= — opcode 102 — emit BYTES (a unibyte
  string or list of 0-255 ints) into =.rodata= and define a LOCAL OBJECT
  symbol NAME (a symbol) at its offset.  Statement-only declaration; emits
  no =.text=.  Parsed in =--parse-stmt=; collected by
  =--collect-data-blobs=; wired into =compile-to-link-unit= (blob bytes
  appended to =.rodata=, =blob-symbols= added to =all-symbols=, blob names
  removed from =extern-names=).
- *No separate =data-addr-local= op was needed.*  The existing
  =(data-addr NAME)= already emits =LEA rax,[rip+disp32]= + a PC32 reloc;
  since =data-blob= registers NAME as a LOCAL defined symbol and filters it
  out of =extern-names=, =data-addr= resolves against the local blob symbol
  automatically.
Two gating fixes were required so the form survives object mode:
- =data-blob= added to =--external-user-call-reserved-ops= (so the
  =parse-value= extern-call fallback never intercepts it), and
- =data-blob= added to the keep-list in =--extract-defmacros= seq handling
  (line ~2166); under =--allow-external-user-calls t= that arm was
  *dropping* every non-=defun=/=prog1= symbol-headed top-level child, which
  silently deleted the blob from =:source=.  This was the real blocker.

Verified end-to-end (=nelisp-aot-compiler/data-blob-*= ERT tests): a blob
+ two reader defuns compile to an ET_REL =.o= where =readelf= shows
=myblob= as =OBJECT LOCAL= in =.rodata= with two =R_X86_64_PC32= relocs; a
C driver links and reads the exact bytes back
(=getb(0)=10 getb(4)=250 getb(7)=200 getword(0)=0x281e140a=).

** Step B — cfront: read-only integer global arrays (the =:deref= bucket)  [DONE 2026-06-24]
Implemented in cfront:
- *Parser* (=nelisp-cfront-parse=): a global declarator now *retains its
  array dimensions* on the type (mirrors local decls via =--fold-dim=),
  instead of parsing and discarding the =[N]= dims.  This was the root of
  the *type half* of =:deref-non-pointer=: a global array typed as a bare
  scalar =long=, so =G[i]= derefed a non-pointer.
- =nelisp-cfront-lower--globals= registry: NAME → (=:type= TY =:bytes=
  UNIBYTE) built by =--collect-globals=, packing const integer
  scalars/arrays little-endian (=--global-bytes= / =--pack-int=); handles
  =int-list= initializers, =char[]= string initializers, and zero-pads to
  the declared dimension.  Non-const / non-integer / struct / pointer
  globals are skipped (left for Step C).
- Globals are appended to each function's =tenv= (shadowed by same-named
  params/locals via =--local-names=) so =type-of= resolves =G=, =G[i]=
  element type, etc.
- =lower-program= emits each collected global as a =(data-blob NAME BYTES
  rodata)=; a global reference lowers through =(data-addr NAME)= — a
  scalar loads (width + sign-extend), an array/struct decays to the
  address (reusing =--rvalue= / =--addr=).  No =data-addr-local= op was
  needed (Step A reuses =data-addr= against the local blob symbol).
- =lower-gap-probe= also binds =--globals= so the metric reflects it.

Verified: ERT =nelisp-cfront-global-*= (collect/bytes unit + data-addr
lowering shape + cc-linked e2e reading unsigned/signed char arrays, an int
array, an int scalar, and a loop summing a global array).  =lower-gap= on
real =sqlite3.pp.c=: =:deref-non-pointer= 110 → 68, overall coverage 75.2%
→ 76.5% (1798 → 1829 / 2392).  Remaining global =:deref= cases need Step C
(struct-member-of-global, pointer globals, non-const array dims).

** Step C — general globals (.data read/write, struct/scalar globals)
- =:unknown-struct= globals (=sqlite3Stat.nowValue[op]=): emit the global
  struct into =.data=/=.bss=; member/index via =data-addr-local= + offset.
- Mutable globals: same address op, normal load/store.

** Step D — string literals (=:string-literal=, 282)  [DONE 2026-06-24]
Implemented in cfront (=nelisp-cfront-lower=):
- =--string-pool= interns each distinct (=equal=) C string literal into a
  unique =nlcf_str_N= symbol (=--intern-string=); filled as a side effect
  of lowering function bodies.
- =--expr= =('str)= lowers to =(data-addr SYM)= (was
  =:string-literal-unsupported=).
- =lower-program= emits one NUL-terminated =(data-blob SYM BYTES rodata)=
  per distinct string (=--string-bytes= = UTF-8 + =\0=), beside the Step B
  global blobs.  =lower-gap-probe= binds the pool too.
- No new grammar op (reuses Step A =data-addr= against the local blob
  symbol); equal literals dedup to one symbol (verified by pointer
  identity in the e2e).

Verified: ERT =nelisp-cfront-string-*= (intern/dedup/NUL unit + cc-linked
e2e: content, empty string, indexing a literal, pointer-identity dedup;
69/69 suite green).  =lower-gap= on real =sqlite3.pp.c=:
=:string-literal-unsupported= 282 → 0, overall coverage 76.5% → *86.7%*
(1829 → 2073 / 2392) — the single largest milestone jump.

* §4 Status / estimate

Feasibility *verified* (§2).  Step A is a focused upstream addition
(object-mode data sections + local symbol + reloc) — moderate but deep
(touches the ET_REL section assembly + symtab + reloc tables).  Steps B-D
are then cfront-side, each re-measured against =lower-gap=.  Expected
combined impact: the ≈480 global/string failures, i.e. coverage from 75%
toward ~95%.  This is the clear next M5 milestone, scoped here so it can
be executed without re-deriving the analysis.
