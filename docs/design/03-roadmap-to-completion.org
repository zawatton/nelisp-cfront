* §0 What "completion" means (honest scope)

"Complete nelisp-cfront" has two defensible targets, ~10-50x apart in
effort:

- *MVP-complete* (=v0.1=): a real C front-end that compiles a *useful C
  subset* (functions, int/long/char/pointer/struct/array, arithmetic,
  control flow, locals, calls, a small libc) end-to-end to native,
  driven from a =.c= file.  Genuinely usable; achievable in a bounded
  number of milestones.
- *library-complete* (=v1.0=): compiles real C libraries and drops their
  host/Rust dependencies for pure-grammar builds.  Two targets:
  - *SQLite* (M5) — replace the rusqlite FFI in =nelisp-sqlite= (the
    original 脱ライブラリ goal).
  - *libxml2* (M6) — replace nelisp-http's host =libxml-parse-html-region=
    dependency (and its lossy regex fallback) with a pure-grammar parser.
  Both are multi-month long-poles behind the hard-tail C features
  (varargs / function pointers / setjmp); libxml2 adds encodings/iconv +
  a multi-file build surface.

Plan: drive to MVP-complete first (M2+M3), then take the hard tail
(M4), then the library targets (M5 SQLite, M6 libxml2).  Each milestone
has a runnable "definition of done" so progress is verifiable, not
asserted.

* §2 Grammar polish backlog (small, from Doc 01/02)

- native =ptr-read/write-u16/u32= ops (currently composed from u8; perf).
- no-param non-foldable-init helper for frame-slot locals (Doc 02 §3):
  pick one of =(opaque K)= grammar op / hidden synthetic param / module
  global seed.
- object-mode rodata/string path (v1 forbids strings in bodies; needed
  for C string literals beyond driver-supplied buffers).

* §3 Operating rules

- Every milestone ships with a runnable check (ERT and/or a =make=
  target) — keep =make test= + the stage targets green.
- Verify each grammar op against nelisp source before relying on it
  (drift rule).  Reuse nelisp (=nelisp-aot-compile-to-object=, allocator,
  syscalls); never fork it.
- Record progress via =anvil-worklog=; design changes as numbered docs.
- Honest status only: "runs" means a native binary produced the asserted
  output, not "should work".
